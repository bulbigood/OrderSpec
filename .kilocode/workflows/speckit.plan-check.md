## Role In The Pipeline

A **per-stage gate** for one document: `plan.md`. It runs after `/speckit.plan` and answers:

> **Is plan.md a correct, complete physical mapping of the spec.md contract onto the repository — at this moment of generation?**

`plan` references the physical world (paths, versions, modules), so this gate **does** read the repository — but only to judge whether the plan was *built correctly* against the repo it was generated from. It does **not** ask whether the repo has *since drifted*; that is `/speckit.analyze`'s temporal question.

**Division of responsibility:**
- **Authoring plan content is owned by `/speckit.plan`**, not by this gate. To add/change a mechanism, location, or stack choice, the user invokes `/speckit.plan "<what they want>"`.
- **This gate is a pure inspector.** It may perform only **mechanical / meaning-preserving** auto-fixes (annotations, verbatim-spec→ID reference, an unambiguous path typo). Anything touching mechanism choice, stack, scope, or contract is surfaced as a **Routing block** naming the exact command that resolves it.

Two output channels only: **Auto-Fixed** (mechanical) and **Routing Required** (to `/speckit.plan`, or `/speckit.specify` when the root is a spec defect).

Boundaries:
- Path existence, `[NEW]`/`[MOD]` correctness, plan structure annotations → `validate-traceability.sh`. Trust it.
- spec self-consistency, glossary, AC form, spec's no-physical-detail purity → `spec-check`.
- tasks ordering / coverage-by-task → `tasks-check`.
- Temporal drift, repo-staleness over time, whole-system constitution sweep, cross-artifact contradictions → `/speckit.analyze`.

## When to Run

Conditional. Recommended after every `/speckit.plan` run (including refinement runs), especially when plan.md was generated by a **weaker model**, spec.md was edited shortly before planning, or you want to catch incompleteness/role-leakage before it cascades.

Can be wired as an automatic post-plan hook (`hooks.after_plan`).

## Goal

Act as an **independent inspector with fresh context** over `plan.md`. Ignore any self-reported checklist inside it. Determine whether plan.md is a **complete, faithful, role-pure physical mapping of spec.md**:

- **Completeness**: every spec obligation (REQ / AC / EDGE / INV / NFR / CON) has a corresponding Mechanism Decision.
- **Mechanism adequacy**: each mechanism actually addresses the obligation, not a name-drop.
- **Role-purity**: plan adds physical detail and does NOT restate spec verbatim or re-draw its diagrams.
- **Physical grounding**: paths and annotations are correct against the repo as generated.
- **Stack consistency**: technologies are mutually compatible and do not violate a spec `CON`.

Fix only mechanical / meaning-preserving defects. Route everything else.

## Out of Scope (do NOT do here)

- **Authoring plan content** — choosing/filling a mechanism, location, or stack. That is `/speckit.plan`'s job; the gate only routes to it.
- Requirement quality / spec self-consistency / glossary / AC form / spec purity → `spec-check`.
- Task ordering, E-M-C, coverage-by-task, SC-to-task → `tasks-check`.
- Repo drift over time, whole-system constitution sweep, cross-artifact contradictions → `/speckit.analyze`.
- Mechanical path counting, dangling-ID detection, annotation presence → the script.

If the root defect lives **in spec** (a contradictory/unimplementable REQ, a missing AC), do NOT distort plan around it. Route it (`P7`) to `/speckit.specify` — the plan cannot compensate for a spec gap.

## Operating Constraints

- **INSPECTOR + MECHANIC**: you MAY modify `plan.md` ONLY for mechanical / meaning-preserving auto-fixes. You MUST NOT author or alter mechanism/stack/contract content, and you MUST NOT edit spec.md or tasks.md.
- **Auto-fix vs Route boundary** (the single, tight write permission):
  - **Auto-fix** ONLY when ALL hold: mechanical or strictly meaning-preserving, exactly one valid correction, no change to contract/scope, obvious/reversible. Examples: a verbatim spec restatement replaced by an ID reference; a missing `[NEW]`/`[MOD]` annotation on an otherwise-correct path; a mechanism keyed to the wrong but obvious ID; a typo'd path that unambiguously resolves. Apply in place, record in **Auto-Fixed**.
  - **Route** for everything else — a missing or inadequate mechanism, a stack choice, a `CON` violation, smuggled-in behavior, a misplaced `[NEW]`. Emit a **Routing block** with a ready-to-run command. The gate does NOT author the resolution.
  - **When in doubt, Route — never Auto-fix.** Choosing or inventing a mechanism is never deterministic; it is always a Route.
- **Constitution authority**: a direct plan-stack-vs-constitution conflict is CRITICAL → Route (the principle is fixed; the compliant route is a plan decision). Full whole-system sweep is `/speckit.analyze`'s.
- **No duplication of mechanical work**: trust the script's path/annotation findings.
- **Generation-baseline stance**: read the repo as the baseline the plan was built against; never reason about future drift — that is `/speckit.analyze`.

## Routing Block Format

When a finding cannot be auto-fixed, emit exactly this; **batch all routing blocks** at the end:

```
### Routing Required: {short title}

**Finding**: {what is wrong or missing}
**Location**: {spec ID / mechanism / plan §section}
**Why owner, not gate**: {touches mechanism/stack/scope/contract — must go through the artifact's author}
**Impact if unresolved**: {what breaks downstream}
**Suggested direction**: {1–2 candidate resolutions; advisory only}
**Run**: `/speckit.plan "{ready-to-run request}"`  (or `/speckit.specify "..."` if the root is a spec defect)
```

The `Run` line is a recommendation to the owner, not an action the gate performs.

## Execution Steps

### 1. Initialize

Run `.specify/scripts/bash/check-prerequisites.sh --json --require-tasks --include-tasks` once from repo root; parse FEATURE_DIR and AVAILABLE_DOCS. Derive: SPEC = `FEATURE_DIR/spec.md`, PLAN = `FEATURE_DIR/plan.md`. tasks.md may be absent; do not require it. Abort with instructions if SPEC or PLAN is missing. For single quotes in args use `'I'\''m Groot'` or double quotes.

### 2. Mechanical Validation (script)

Run:

```bash
.specify/scripts/bash/validate-traceability.sh --json "$FEATURE_DIR"
```

- Parse `summary`, `inventory`, `coverage`, `findings` (IDs `M1`–`M14`).
- Import **only plan-relevant** findings (path existence M8, `[NEW]`/`[MOD]` correctness M10, plan annotations M9, plan-side dangling IDs). Keep IDs/severities. Ignore tasks-ordering and timestamp findings.
- You MAY escalate severity with justification (a `[MOD]` path missing for a P1 mechanism → CRITICAL).
- A missing/incorrect annotation on an otherwise-correct path (M9) → **Auto-fix**. A missing path (M10) → **Route** (file may need creating, path may be wrong, or mechanism may need rethinking).
- **Fallback**: if the script is missing/fails, emit HIGH `P0-001` ("mechanical validator unavailable — degraded mode") and spot-check only: each `[NEW]`/`[MOD]` path plausibly resolves; the Mechanism Decisions table is present and ID-keyed. Keep it brief.

### 3. Semantic Detection Passes (LLM — plan meaning, DETECT ONLY)

Load spec and plan; read the repo for grounding spot-checks. Build minimal internal models — do not dump raw artifacts. Limit to 30 findings total (including M-findings); on overflow drop LOW first, aggregate into one line — never drop a CRITICAL. Stable IDs per pass (P1-001…). Classify each as **Auto-fix**, **Route**, or **defer** (spec-rooted → Route to specify).

#### P1. Completeness

- P1a: For each spec obligation (REQ/AC/EDGE/INV/NFR/CON), confirm a Mechanism Decision exists keyed by ID. Any obligation with **no** mechanism → **Route** (choosing one is design — owned by plan). Never invent silently.
- P1b: A missing mechanism for a **P1-story** obligation (spec §14) is CRITICAL; non-MVP is HIGH/MEDIUM by impact. Aggregate large gaps into one Routing block plus the worst specifics.
- P1c: `CON` and `NFR` are easy to forget — confirm each has a mechanism or an explicit, justified deferral. Missing → **Route**.

#### P2. Mechanism Adequacy

- P2a: Each Mechanism Decision **actually addresses** its obligation, not just name-drops the ID. An inadequate/empty mechanism → **Route**.
- P2b: A mechanism with a measurable NFR claim (latency/throughput/limits) is **plausibly capable**. Clearly implausible → **Route**.

#### P3. Role-Purity

- P3a: plan does NOT restate spec verbatim or re-draw its ERD/contract diagrams → **Auto-fix** (replace the restatement with a spec ID reference, preserving any physical detail around it).
- P3b: plan DOES carry physical detail (paths, modules, library/version, `[NEW]`/`[MOD]`). A plan abstract where it must be concrete → **Route** (the concrete choice is a plan decision). HIGH.
- P3c: plan does NOT smuggle in product/contract decisions absent from spec (new behavior, new AC) → **Route** (`P7`) to `/speckit.specify`: promote into spec, or drop from plan.

#### P4. Physical Grounding (beyond the script)

- P4a: For a `[MOD]` file that exists, does plan assume a function/class/export actually present? Sample at most 2–3 highest-risk claims. A wrong assumption → **Route**.
- P4b: `[NEW]` paths do not collide and match repo structure. A misplaced `[NEW]` with one obviously-correct conventional location → **Auto-fix** (relocate); ambiguous → **Route**.

#### P5. Stack Consistency

- P5a: Technologies are mutually compatible (no contradictory framework/runtime across mechanisms). A genuine contradiction → **Route**.
- P5b: The stack does not violate a spec `CON`. `CON` mandates X, plan chose Y → **Route** (switch to X, or challenge the CON via `/speckit.specify`). HIGH (CRITICAL if it also breaks a constitution MUST).

#### P6. Direct Constitution Conflicts (narrow)

- P6a: Flag only **direct** plan-stack-vs-constitution conflicts (forbidden dependency, violated architectural MUST). CRITICAL → **Route**. Do NOT perform the full sweep — that is `/speckit.analyze`.

### 4. Severity Assignment

- **CRITICAL**: a P1/MVP obligation has no mechanism; plan stack violates a constitution MUST; a `[MOD]` path missing for a P1 mechanism.
- **HIGH**: a non-MVP obligation has no mechanism; stack violates a spec `CON`; plan abstract where physical detail is mandatory (P3b); inadequate mechanism for a P1 obligation.
- **MEDIUM**: verbatim spec duplication (P3a, usually auto-fixed); non-MVP inadequate mechanism; `[NEW]` placement inconsistent with repo; spec-rooted reroute (`P7-*`).
- **LOW**: cosmetic duplication; minor annotation omissions; non-contract-bearing restatement.

**MVP-scope**: obligations covered by stories whose UJ priority is P1 in spec §14. A HIGH on P1-scope blocks; the same class on non-MVP does not auto-block.

### 6. Next Actions

The gate's responsibilities end at detection, mechanical repair, and routing:
- **Mechanical / meaning-preserving** → already auto-fixed; no user action.
- **Mechanism / stack / scope** → see the Routing block; resolve via `/speckit.plan "..."`. The gate did not write this content.
- **Spec-rooted** (`P7`) → resolve via `/speckit.specify "..."`; plan never compensates for a spec gap.

Recommended loop: run the routed commands (batch the requests if convenient), then **re-run `/speckit.plan-check`** to confirm. Idempotent: a plan with no design/contract findings yields ✅ PASS. Repo-drift-over-time or whole-system concerns → `/speckit.analyze`.

## Operating Principles

- **Pure inspector for content**: the gate detects and routes; it NEVER authors a mechanism, location, or stack choice. Filling plan content is `/speckit.plan`'s job.
- **One narrow write permission**: auto-fix applies ONLY to mechanical / strictly meaning-preserving defects with a single obvious correction. When in doubt, Route.
- **Route, don't ask-and-apply**: design/contract findings become Routing blocks with a ready-to-run command — the gate never asks a content question whose answer it would write.
- **Generation-baseline stance**: validate against spec and the repo-as-baseline; never reason about future drift.
- **NEVER hallucinate missing sections** — an absent mechanism is the core finding type here, reported as a Route.
- Trust a passed `spec-check`; a spec defect is routed to `/speckit.specify`, never patched into plan.
- Mechanics belong to the script; **only plan meaning (completeness, adequacy, role, stack) belongs to your LLM tokens**.
- Minimal high-signal tokens: cite specific spec IDs and mechanisms, cap at 30 findings, aggregate large gaps.
- A missing mechanism for a P1 obligation is always CRITICAL; a stack choice violating a constitution MUST is always CRITICAL — both routed, both BLOCK.

## Role In The Pipeline

A **per-stage gate** for one document: `tasks.md`. It runs after `/order.tasks` and answers:

> **Is tasks.md a correct, faithful ordering of decisions already made in plan.md — at this moment of generation?**

It is **local** by design: narrow context (spec for the ID vocabulary, plan as the decision source, tasks as the subject). It does **not** read the repository and does **not** reason about time — cross-stage concerns owned by `/order.analyze`.

**Division of responsibility:**
- **Authoring task content is owned by `/order.tasks`**, not by this gate. To add a task, fill a true coverage gap, or change what a task does, the user invokes `/order.tasks "<what they want>"`.
- **This gate is a pure inspector.** It may perform only **mechanical / structural meaning-preserving** auto-fixes — these are defined by the E-M-C / test-first methodology, not by content choice (reordering, inserting a required GATE/verification task, fixing tags, numbering). Anything that **creates or changes the content of a task, or fills a true coverage gap**, is surfaced as a **Routing block**.

Two output channels only: **Auto-Fixed** (mechanical/structural) and **Routing Required** (to `/order.tasks`, or `/order.plan` / `/order.spec` when the root is upstream).

Boundaries:
- Coverage counts, ID integrity, numbering, `[P]` file-disjointness → `validate-traceability.sh`. Trust it; do not re-count.
- Upstream validity → `spec-check` (spec self-consistency) and `plan-check` (plan↔spec completeness, plan role-purity).
- Temporal drift / repo-staleness / whole-system constitution / cross-artifact contradictions → `/order.analyze`.

## When to Run

Conditional. Recommended after every `/order.tasks` run (including refinement runs), especially when tasks.md was generated by a **weaker model**, plan.md was edited shortly before, or you want to catch ordering/coverage defects before they cascade into `/order.code`.

Can be wired as an automatic post-tasks hook (`hooks.after_tasks`).

## Goal

Act as an **independent inspector with fresh context** over `tasks.md`. Ignore any self-reported checklist inside it. Determine whether tasks.md is a **faithful, well-ordered, complete-coverage projection of plan.md**:

- **No new decisions**: tasks only sequence decisions already in spec/plan.
- **Coverage**: every spec ID that needs a task has one (mechanical via script; semantic gaps via LLM).
- **E-M-C ordering**: Expand → Migrate/Implement → Contract discipline holds.
- **Test-first discipline**: within a story, test tasks precede implementation; a verification task closes the phase.
- **SC buildability**: success criteria requiring buildable work are reflected in tasks.
- **Task format**: each task is actionable, correctly tagged, file-scoped.

Fix only mechanical / structural meaning-preserving defects. Route everything that creates or changes task content.

## Out of Scope (do NOT do here)

- **Authoring task content** — adding a task, filling a true coverage gap, deciding how an SC is realized. That is `/order.tasks`'s job; the gate only routes to it.
- Requirement quality / spec self-consistency / glossary / AC form → `spec-check`.
- plan↔spec completeness, plan role-purity, plan duplication of spec → `plan-check`.
- Repository state, `[NEW]`/`[MOD]` path existence, temporal drift, constitution alignment, cross-artifact contradictions → `/order.analyze`.
- Mechanical ID counting, dangling-ID detection, raw `[P]` file-disjointness → the script.

If the root defect lives **upstream** (a wrong plan Mechanism, a spec ambiguity), do NOT patch tasks around it. Route it (`T7`) to `/order.plan` or `/order.spec`.

## Operating Constraints

- **INSPECTOR + MECHANIC**: you MAY modify `tasks.md` ONLY for mechanical / structural meaning-preserving auto-fixes. You MUST NOT author task content or fill a true coverage gap, and you MUST NOT edit plan.md or spec.md.
- **Auto-fix vs Route boundary** (the single, tight write permission):
  - **Auto-fix** ONLY when ALL hold: the correction is mechanical or structural-per-methodology, exactly one valid form exists, it does NOT change contract/scope, and it is obvious/reversible. Examples: ID-reference typo, task numbering, test/impl ordering swap, inserting a required per-phase verification task, inserting a required Contract GATE, moving a destructive step after the GATE when placement is unambiguous, fixing a `[US#]` tag resolvable from context, removing a `[P]` whose description depends on an adjacent `[P]`, terminology-to-glossary. Apply in place, record in **Auto-Fixed**.
  - **Route** for everything else — a task that invents a decision, a true coverage gap with no home task, a vague task whose target is not unambiguous in plan, an SC needing a new realizing task, a cross-story dependency. Emit a **Routing block** with a ready-to-run command.
  - **When in doubt, Route — never Auto-fix.** Creating or changing what a task *does* is never deterministic.
- **No duplication of mechanical work**: trust the script's coverage/format/`[P]` findings.
- **Local stance**: do not reach into the repository or reason about time — that is `/order.analyze`.

## Routing Block Format

When a finding cannot be auto-fixed, emit exactly this; **batch all routing blocks** at the end:

```
### Routing Required: {short title}

**Finding**: {what is wrong or missing}
**Location**: {task IDs / section}
**Why owner, not gate**: {creates/changes task content or coverage — must go through the artifact's author}
**Impact if unresolved**: {what breaks downstream}
**Suggested direction**: {1–2 candidate resolutions; advisory only}
**Run**: `/order.tasks "{ready-to-run request}"`  (or `/order.plan "..."` / `/order.spec "..."` if the root is upstream)
```

The `Run` line is a recommendation to the owner, not an action the gate performs.

## Execution Steps

### 1. Initialize

Run `.orderspec/scripts/bash/check-prerequisites.sh --json --require-tasks --include-tasks` once from repo root; parse FEATURE_DIR and AVAILABLE_DOCS. Derive: SPEC = `FEATURE_DIR/spec.md`, PLAN = `FEATURE_DIR/plan.md`, TASKS = `FEATURE_DIR/tasks.md`. Abort with instructions if any required file is missing. For single quotes in args use `'I'\''m Groot'` or double quotes.

### 2. Mechanical Validation (script)

Run:

```bash
.orderspec/scripts/bash/validate-traceability.sh --json "$FEATURE_DIR"
```

- Parse `summary`, `inventory`, `coverage`, `findings` (IDs `M1`–`M14`).
- Import **only tasks-relevant** findings (coverage gaps, task numbering, `[P]` disjointness, dangling task IDs). Keep IDs/severities. Ignore plan/repo/timestamp findings.
- You MAY escalate severity with justification (an uncovered AC of the MVP/P1 story → CRITICAL).
- **Fallback**: if the script is missing/fails, emit HIGH `T0-001` ("mechanical validator unavailable — degraded mode") and spot-check only: each P1 AC maps to ≥1 task; task IDs sequential/unique; no two `[P]` on the same path. Keep it brief.

### 3. Semantic Detection Passes (LLM — task intent, DETECT ONLY)

Load spec, plan, tasks. Build minimal internal models — do not dump raw artifacts. Limit to 25 findings total (including M-findings); on overflow drop LOW first, aggregate into one line — never drop a CRITICAL. Stable IDs per pass (T1-001…). Classify each as **Auto-fix**, **Route**, or **defer** (upstream → Route to plan/spec).

#### T1. No New Decisions (faithfulness to plan)

- T1a: Any task introducing a design decision **absent from spec and plan** (file, mechanism, library, schema field, endpoint not derivable upstream) → **Route** (legitimize it via `/order.plan`, or strip it via `/order.tasks`).
- T1b: Any task referencing a plan Mechanism / file / path **not in the current plan**: an obvious typo for an existing mechanism → **Auto-fix** (correct the reference); something genuinely absent → **Route**.

#### T2. Coverage Semantics (gaps the script cannot see)

- T2a: For each spec ID the script marks "covered", does the task **actually address it** or just name-drop the ID? Task implements the behavior but omitted the ID tag → **Auto-fix** (add the tag). No task does the work → **Route** (true coverage gap).
- T2b: EDGE / INV needing executable behavior backed by a task (pure-documentation invariants exempt). Missing coverage → **Route** (the work doesn't exist; authoring it is `/order.tasks`).

#### T3. E-M-C Ordering

- T3a: Phase 1 (Expand) tasks are **additive only** (read descriptions, not just labels). A destructive step in Expand → **Auto-fix** (move to the correct phase) when unambiguous; else **Route**.
- T3b: Story phases follow UJ priority order from spec §14 (P1 before P2). Out-of-order with a single correct ordering → **Auto-fix** (reorder).
- T3c: The Contract phase begins with a GATE task; **no destructive task** precedes that GATE. Missing GATE → **Auto-fix** (insert it). A destructive task ahead of it → **Auto-fix** (move after GATE) if placement is unambiguous; else **Route**.

#### T4. Test-First Discipline

- T4a: Within each story phase, test tasks **precede** implementation (classify by description). A clear test-after-impl pair → **Auto-fix** (swap order).
- T4b: Each story phase is **closed by a verification task** asserting the story's ACs. Missing → **Auto-fix** (append one referencing that story's AC IDs).
- T4c: Test tasks state the expectation to **fail first** (red). Missing red-state note → **Auto-fix** (add it) where the test's intent is clear.

#### T5. SC Buildability

- T5a: Each Success Criterion (spec §2) implying **buildable work** (load tests, security tooling, performance assertions) is reflected by ≥1 task; post-launch / business KPIs exempt. A buildable SC with no task → **Route** (the realizing task is content owned by `/order.tasks`).

#### T6. Task Format

- T6a: Each task is **actionable and self-contained**: exact file path + spec IDs + short paraphrase. A vague task ("handle errors") → **Auto-fix** by enriching from plan when the target is unambiguous in plan; else **Route**.
- T6b: `[US#]` tags trace each implementation task to a story; `[P]` semantics respected at description level. Missing `[US#]` resolvable from context → **Auto-fix**. A `[P]` whose description depends on an adjacent `[P]`'s output → **Auto-fix** (remove the offending `[P]`). NOTE: **absence of `[P]` is NEVER a finding** — sequential is always valid; never add `[P]`.
- T6c: Cross-story dependency breaking independent-testability (a P1 task depending on a P2 task) → **Route** (re-sequencing stories touches the delivery contract).

#### T7. Upstream Reroute (not a pass)

If you notice an upstream defect (a wrong plan Mechanism, a spec ambiguity, requirement-quality issues): do NOT patch tasks and do NOT edit upstream from here. Emit a Routing block to `/order.plan` or `/order.spec` describing the suspected root.

### 4. Severity Assignment

- **CRITICAL**: a P1/MVP AC has no task (true coverage gap); a destructive task precedes the Contract GATE and placement is ambiguous; a task invents a decision that changes MVP scope.
- **HIGH**: tests ordered after implementation within a story; a story phase with no verification task; a task references a non-existent plan mechanism; SC requiring buildable work with no task.
- **MEDIUM**: non-MVP EDGE/INV uncovered; vague/non-self-contained task; upstream reroute (`T7-*`); cross-story dependency on a non-MVP story.
- **LOW**: cosmetic format issues; minor paraphrase verbosity; redundant ID listing.

**MVP-scope**: stories whose UJ priority is P1 in spec §14. A HIGH on a P1 story blocks; the same class on P2+ does not auto-block.

### 6. Next Actions

The gate's responsibilities end at detection, mechanical/structural repair, and routing:
- **Mechanical / structural** → already auto-fixed; no user action.
- **Content / coverage** → see the Routing block; resolve via `/order.tasks "..."`. The gate did not author this content.
- **Upstream root cause** (`T7`) → resolve via `/order.plan "..."` or `/order.spec "..."`; tasks never compensate for an upstream gap.

Recommended loop: run the routed commands (batch the requests if convenient), then **re-run `/order.tasks-check`** to confirm. Idempotent: a faithful, fully-covered tasks.md yields ✅ PASS. Repo/drift/whole-system concerns → `/order.analyze`.

## Operating Principles

- **Pure inspector for content**: the gate detects, repairs structure, and routes; it NEVER authors a task or fills a true coverage gap. Filling task content is `/order.tasks`'s job.
- **Structural auto-fix is methodology-driven, not content**: reordering, GATE/verification insertion, and tag fixes follow fixed E-M-C / test-first rules — meaning-preserving. Creating or changing what a task *does* is always a Route.
- **One narrow write permission**: when in doubt, Route — never Auto-fix.
- **Route, don't ask-and-apply**: content/coverage findings become Routing blocks with a ready-to-run command.
- **Local-stage stance**: validate tasks.md against plan.md only; never reach into the repo or reason about time.
- **NEVER hallucinate missing sections** — report absences accurately.
- Trust a passed upstream `*-check`; an upstream root cause is routed to its owner, never patched around.
- Mechanics belong to the script; **only task-intent meaning belongs to your LLM tokens**.
- Minimal high-signal tokens: cite specific task IDs, cap at 25 findings, summarize overflow.
- A P1/MVP coverage gap is always at least HIGH; a destructive task before the GATE with ambiguous placement is always CRITICAL.
- Under-parallelization is never a defect; absence of `[P]` needs no remediation.
